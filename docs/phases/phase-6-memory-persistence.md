# Phase 6: 记忆永久化与上下文压缩

## 状态

待开始 ☐

## 详细 Checkpoint

### 方向九. 永久化记忆与上下文压缩 (§14.5)

#### 目标

将当前 Redis TTL 临时存储升级为永久化存储层, 并增加上下文压缩能力支撑长任务和长会话.

- [ ] Checkpoint 14.4.1 记忆永久化存储层
  - 实现项: 新增 MySQL/SQLite 持久层作为 Redis 的落盘后端. 关键数据 (Skills, 高质量 Lessons, 用户偏好, 审批历史) 永久保存. 运行态快照仍用 Redis (高性能), 周期性落盘.
  - 验收方法: 运行记忆持久化集成测试. 模拟 Redis 重启后恢复.
  - 验收证据: 持久化记录. Redis 重启后数据恢复日志. 一致性校验报告.
  - 通过标准: 关键记忆在 Redis 重启后可恢复. 落盘延迟不影响主链性能. 数据一致性 100%.

- [ ] Checkpoint 14.4.2 记忆分级 TTL 策略
  - 实现项: 定义 `ephemeral` (工作态, 24h) `short_term` (任务记忆, 7d) `long_term` (经验, 永久) `permanent` (Skill, 配置, 永久) 四级 TTL. 自动根据 memory_entry 类型分配.
  - 验收方法: 运行 TTL 策略单测和过期清理集成测试.
  - 验收证据: 各级别 TTL 配置. 过期清理日志. 永久记忆未被清理的验证.
  - 通过标准: 各级别 TTL 正确生效. 永久记忆不被误清. 过期清理不影响运行中任务.

- [ ] Checkpoint 14.4.3 上下文压缩器
  - 实现项: 新增 `ContextCompressor`. 在 LLM 调用前估算 token 数, 超限时触发压缩. 策略: 保护头尾消息 + 中间历史 LLM 摘要 + 任务链分段 checkpoint.
  - 验收方法: 运行 2 个长任务链 case (> 20 步) 和 1 个短任务对照 case.
  - 验收证据: 压缩触发日志. 压缩前后 token 数对比. 任务完成质量对比.
  - 通过标准: 长任务链不因上下文超限而失败. 压缩后任务完成质量不低于未压缩短任务的 90%.

- [ ] Checkpoint 14.4.4 会话分段与链接
  - 实现项: 长任务自动分段, 每段有 summary checkpoint. 新段通过 `parent_segment_id` 链接. 恢复时可从任意段的 checkpoint 继续.
  - 验收方法: 运行 1 个超长任务 case 和 1 个从中间段恢复 case.
  - 验收证据: 分段记录. parent_segment_id 链. 恢复日志.
  - 通过标准: 超长任务自动分段且不丢失上下文. 从中间段恢复成功.

## 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|-----|------|------|----------|
| Redis -> DB 迁移期间的数据一致性风险 | 中 | 高 | 双写策略 + 一致性校验 + 回滚机制 |
| 落盘延迟影响主链性能 | 中 | 中 | 异步落盘 + 批量写入 + 只对关键数据落盘 |
| 上下文压缩导致关键信息丢失 | 中 | 高 | 保护头尾消息 + 压缩后质量对比验证 |
| 分段恢复时上下文不连贯 | 低 | 中 | 每段保留完整 summary checkpoint + parent 链接 |

## 成功标准 (Exit Criteria)

- 实现 Redis -> MySQL/SQLite 落盘层
- 实现四级 TTL 分级策略
- 实现上下文压缩器 (preflight + 中间摘要)
- 实现会话分段与链接
- Redis 重启后关键记忆可恢复
- 超长任务 (20+ 步) 可正常完成
