# 架构决策记录 (ADR)

## ADR-001: LLM 调用缓存策略

**日期**: 2026-03-17  
**状态**: Accepted  
**决策者**: 项目团队

### 问题

当前系统 LLM 调用延迟高（P95: 153秒），token 消耗大（238,326 tokens/次），需要优化性能。

### 决策

引入 LLM 调用缓存，使用纯内存 LRU 缓存策略。

### 原因

- temperature=0 的请求是确定性的，可以安全缓存
- 纯内存实现简单，无额外依赖
- LRU 策略可以控制内存使用

### 后果

- 正面：
  - 缓存命中时延迟大幅降低
  - Token 消耗减少
- 负面：
  - 内存占用增加（默认 1000 条）
  - 只对 temperature=0 的请求有效

## ADR-002: 输出自动修复策略

**日期**: 2026-03-17  
**状态**: Accepted  
**决策者**: 项目团队

### 问题

当前契约失败率 26.76%，主要原因是 LLM 输出 JSON 不稳定。

### 决策

实现 3 层自动修复机制：
1. 直接解析 JSON
2. 从文本中提取 JSON（处理 ```json 包裹）
3. 修复常见 JSON 问题（尾随逗号、注释等）

### 原因

- 不需要改变 LLM 调用逻辑
- 兼容现有代码
- 覆盖大多数常见情况

### 后果

- 正面：
  - 契约失败率预期降低到 <10%
  - 自动生效，无需改动 Agent 代码
- 负面：
  - 极端情况下仍可能失败

## ADR-003: 配置管理方案选择

**日期**: 2026-03-17  
**状态**: Accepted  
**决策者**: 项目团队

### 问题

当前配置使用多个独立函数，缺少类型安全，验证困难。

### 决策

使用 Pydantic Settings 进行配置管理，同时保留原 config.py 作为向后兼容层。

### 原因

- Pydantic Settings 提供类型安全
- 支持环境变量和 .env 文件
- 向后兼容，不破坏现有代码

### 后果

- 正面：
  - 类型安全，IDE 提示更好
  - 配置验证自动
  - 新代码可以直接使用
- 负面：
  - 有两套配置方案，过渡期需要维护

## ADR-004: 消息总线实现方案

**日期**: 2026-03-17  
**状态**: Accepted  
**决策者**: 项目团队

### 问题

Agent 之间缺少通信机制，协作指标 IDS 只有 0.0545。

### 决策

使用纯内存实现的消息总线，支持发布/订阅和点对点消息。

### 原因

- 项目已有 LangGraph 共享状态，不需要复杂的分布式消息
- 纯内存实现简单，性能好
- 可以随时扩展到 Redis/Kafka

### 后果

- 正面：
  - 简单快速
  - 无额外依赖
- 负面：
  - 进程重启消息丢失
  - 不支持分布式

## ADR-005: 错误处理策略

**日期**: 2026-03-17  
**状态**: Accepted  
**决策者**: 项目团队

### 问题

当前错误处理过于宽泛，缺少错误分级和死信队列。

### 决策

引入错误分级、可恢复/不可恢复区分、死信队列。

### 原因

- 便于后续集成监控告警
- 可恢复错误可以重试
- 死信队列用于故障排查

### 后果

- 正面：
  - 错误分类更清晰
  - 为后续监控告警做准备
- 负面：
  - 当前版本主要是框架，还没有集成到现有流程
